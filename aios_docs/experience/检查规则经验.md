# 检查规则经验

## 经验 1：确定性检查优先程序化

经验等级：A

适用场景：所有项目的检查层设计。

不适用场景：纯创意、审美、体验类判断。

问题表现：让 LLM 判断文件是否存在、测试是否通过、是否误删文件，结果不稳定。

原因分析：确定性问题不需要语言模型推理，程序检查更可靠。

解决方案：文件、测试、输出、路径、数量、超时、删除风险等用 Python 检查。

以后规则：确定性检查不要交给 LLM 猜。

置信度：高

来源项目：AIOS 初始设计。

日期：2026-06-05

是否可能过期：否

过期检查方式：长期有效。

## 经验 2：判断性检查交给 LLM，但必须有样例

经验等级：A

适用场景：用户体验、专业感、内容质量、自然度、连贯性判断。

不适用场景：文件存在、测试结果、数值阈值等确定性问题。

问题表现：LLM 自信判断“体验很好”，但用户实际不满意。

原因分析：LLM 没有读取规格和正反样例，判断标准漂移。

解决方案：LLM 判断前必须读取规格、验收标准、黄金样例、失败反例。

以后规则：判断性检查必须带 rubric 和 examples。

置信度：高

来源项目：AIOS 初始设计。

日期：2026-06-05

是否可能过期：否

过期检查方式：长期有效。
